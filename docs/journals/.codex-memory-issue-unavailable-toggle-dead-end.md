## 배경

whitepaper는 popup을 `On/Off toggle + short status line`으로 정의하고, selector failure 후에는 `Unavailable`을 보여 주되 다음 refresh 또는 chat navigation에서 fresh init이 가능해야 한다고 설명합니다. 현재 popup은 `Unavailable`일 때 토글 자체를 비활성화합니다.

## 근거

- `whitepaper.md:183-205`
- `src/popup-view.ts:9-16`은 `status === "Unavailable"`이면 `disabled: true`를 반환합니다.
- `src/popup.ts:42-48`은 이 view model을 그대로 적용해 토글을 비활성화합니다.
- `tests/unit/popup-worker.test.ts:387-398`도 `Unavailable`이면 disabled toggle을 기대합니다.
- `docs/chrome-extension-loading.md:98-101`은 `Unavailable` 상태에서 탭을 새로고침하라고 안내합니다.

## 문제

현재 popup에서 의도적으로 refresh를 일으키는 1차 제어는 On/Off 토글입니다. 그런데 `Unavailable`에서 토글이 막히면 popup 자체로는 recovery를 시도할 수 없고, 사용자는 수동 브라우저 refresh에 의존해야 합니다. 이는 popup control model과 recovery 설명이 어긋나는 상태입니다.

## 기대 동작

- `Unavailable`에서도 토글은 계속 보여야 합니다.
- 사용자가 Off/On을 통해 refresh 기반 recovery를 시도할 수 있어야 합니다.
- 저장된 preference는 whitepaper대로 유지하되, UI가 dead-end가 되면 안 됩니다.

## 완료 조건

- `Unavailable` 상태가 popup dead-end가 되지 않습니다.
- toggle + refresh 경로로 recovery를 시도할 수 있습니다.
- popup unit/integration 테스트가 이 동작을 검증합니다.
